Part 1

Setup

  • start eclipse.exe with -consolelog -data X, where X is a new workspace i.e. points to not yet existent directory
Note: Keep an eye on the console for error messages,

Java Project Creation & Source Import

  • open the Java preferences and check on the Installed JREs page that a JRE has been detected and that this JRE's system libraries were correctly detected as well. Edit the installed JRE and check if the rt.jar has its source attached. If this is not the case install a Java SDK which contains the source and use it as default JRE.
  • create a new Java Project JUnit
    • click through the wizard and use the default settings
    • verify:
      • the only open perspective is the Java perspective
      • there is only the JUnit project and the JRE System Library - no source folder and no bin folder
      • the Java Build Path by opening the properties page of the JUnit project:
        • the Java build path order should be JUnit project followed by the JRE System Library

  • this smoke test uses JUnit source. JUnit is an open source unit testing framework for Java. Refer to http://www.junit.org/ for more information on JUnit.
    Import Junit 3.8.1 source into your JUnit project:

    • download the junit3.8.1src.zip (143 kb) to your computer
    • in the Package Explorer view select the JUnit project
    • from the context menu click Import...
    • select General > Archive file
    • click Next >
    • click Browse... to select the zip file
    • select the junit3.8.1src.zip that you downloaded to your computer before
    • click Finish

Editing

  • switch back to the Java perspective
  • open junit.framework.TestCase.java from the Package Explorer
  • verify that the
    • Java editor with TestCase.java is active and has focus
    • Outline view is populated
    • Outline view shows the decorators for abstract, constructor, implemented and overridden methods
    • syntax coloring in the Java editor is OK
  • select a method in the Outline view
    • verify that it is revealed in the Java editor and the magnet shows up
  • use Java Content Assist on some type argument
  • use Javadoc Content Assist inside a java doc comment to insert @see TestResult#addFailure(Test, AssertionFailedError)
  • select the run method in the body of run(TestResult result):

        result.run(this);
      }
  • hover over run inside the above method and verify that javadoc help is shown in a tool tip
  • code resolve
    • press F3 or execute Open on selection
    • TestResult.java should open
  • test reconciling
    • add a method
      • verify that it shows up in the Outline view
    • add a syntax error
      • verify that the syntax error is underlined in red
      • remove the syntax error
    • select the new method in the Outline view and select Delete from the context menu
  • use the actions
    • Source > Shift Right
    • Source > Shift Left
    • Edit > Find/Replace...
    • Edit > Undo
    • File > Revert
  • switch to the segmented view by clicking the Show Source of Selected Element Only button in the tool bar
    • select methods in the Outline view and verify that the corresponding source is shown
  • test local history

    • save it
    • select the changed method in the Outline view

    • select the previous edition and verify that it gets replaced properly
  • organize import

  • generate getter/setter

    • verify that the one getter an one setter have been created

Building

  • rebuild the JUnit project (Project > Clean...) and choose Clean all projects
    • verify in the Navigator view that there are classes with each Java source file (in the same folders)
    • no errors should show up in the Problems view

  • because auto-build is on (default) a build should be executed when the resource is saved
    • the errors should show up in the Problems view
    • error ticks should show up in the Package Explorer view
    • error marker should show up in the editor
      • verify that the error message is shown in hover when moving the mouse over the marker
    • an error tick should show up in the editor tab of TestCase.java
  • in the Problems view use Go To action (or double-click) to navigate to the Java editor
  • fix the error and save again (auto-build should start a build)
    • the error tasks should disappear
    • the error ticks in the Package Explorer view should disappear
    • error marker should disappear in editor

    Java Resource Compare

    • disable auto-build
    • in Navigator view drill down to JUnit/junit/runner folder
    • copy the folder runner into folder junit (name it runner2)
    • compare both folders (context menu: Compare With > Each Other)
      • Information dialog: There are no differences between the selected inputs
    • in folder runner2 delete file Sorter.java
    • in runner open TestCaseClassLoader.java and make these changes:
      • add a new method void foo(){}
      • delete method getResource(String)
      • change the body of the method isExcluded(String) by deleting a line
    • save the file
    • compare both folders runner and runner2 (context menu: Compare With > Each Other)
      • the resulting compare editor should list two changed resources:
        • an addition of Sorter.java to runner
        • a change of TestCaseClassLoader.java
    • in the top left pane of the compare editor double-click on TestCaseClassLoader.java
      • a textual compare viewer shows TestCaseClassLoader.java side by side
      • a new viewer opens on the right hand side and shows the structural differences of your changes
        • added method foo()
        • removed method getResource(String), and
        • changed method isExcluded(String)
      • verify your modification by stepping through all changes with the "down arrow" button or by pressing the Control + . shortcut
    • select method isExcluded(String)
      • verify that the selected element appears in a textual compare viewer in the bottom pane

    • delete the runner2 folder
